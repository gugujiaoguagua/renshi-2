# 规则总表

## 规则等级

| 等级 | 含义 | 处理方式 |
|---|---|---|
| P0 | 硬规则，绝对不能违反 | 违反则停止并说明 |
| P1 | 项目规则，默认必须遵守 | 特殊情况需要说明 |
| P2 | 流程规则，推荐遵守 | 可按任务调整 |
| P3 | 临时规则，仅当前任务有效 | 复盘后决定是否升级 |

## 默认规则

| 编号 | 等级 | 规则 | 适用范围 | 状态 |
|---|---|---|---|---|
| R-GEN-001 | P0 | 不得编造不存在的业务字段、文件、数据、结论 | 全项目 | 生效 |
| R-GEN-002 | P0 | 不得覆盖、删除或改写原始资料，除非用户明确要求 | 文件处理 | 生效 |
| R-GEN-003 | P1 | 每次任务开始应创建或更新任务卡 | 任务管理 | 生效 |
| R-GEN-004 | P1 | 每次任务结束应写复盘记录 | 资产沉淀 | 生效 |
| R-GEN-005 | P1 | 新发现的业务规则先进入 memory 或升级记录，不直接写入正式 skill | 规则升级 | 生效 |
| R-GEN-006 | P2 | 同一规则至少经过两次任务验证后，才建议进入规则总表或 SKILL.md | 规则升级 | 生效 |
| R-GEN-007 | P2 | 输出前应按质量检查清单自检 | 交付验收 | 生效 |
| R-GEN-008 | P2 | 高频、多人、复杂状态流转的任务应进入应用化候选池 | 应用化判断 | 生效 |
| R-CODE-001 | P1 | 进入 demo 或应用开发前，必须先确定 AI 友好代码架构 | 代码开发 | 生效 |
| R-CODE-002 | P1 | 业务规则不得写死在 UI 事件中，应放入 rules 或独立文档 | 代码开发 | 生效 |
| R-CODE-003 | P1 | 数据访问必须通过 services/api 层封装，页面不得直接散落调用底层接口 | 代码开发 | 生效 |
| R-CODE-004 | P2 | 新增字段应先更新 types，再更新服务、页面和测试 | 代码开发 | 生效 |
| R-CODE-005 | P2 | AI 修改代码前应先确认模块边界、规则文件和类型定义 | 代码开发 | 生效 |
