# AI 友好代码架构规范

## 目的

当某个流程从 skill 进入应用化、demo 或工具开发阶段时，必须使用 AI 友好代码架构。目标是让 AI 每次只需要理解一个清晰范围，就能安全修改、扩展和验证。

核心原则：

```text
让 AI 每次只改一个模块、一个规则、一个服务或一个页面。
```

## 适用场景

- 从工作流资产包进入 demo 搭建。
- 从 skill 进入小工具开发。
- 从业务需求进入 Web / App / 后台系统开发。
- 需要 Claude / Codex 持续参与迭代的项目。

## 总体原则

1. 按业务模块组织，不按零散技术文件堆放。
2. 页面、服务、规则、类型、工具函数分开。
3. 业务规则不能藏在 UI 事件里。
4. 数据访问必须统一封装。
5. 新增字段先改类型，再改服务和页面。
6. 新增流程先写任务卡或需求说明，再写代码。
7. 每个核心业务模块必须有模块说明。
8. AI 修改代码前，先读取对应模块说明、规则和类型。
9. 高频复杂流程先用 skill 跑通，再应用化。

## 推荐前端项目结构

适用于 Angular / React / Vue 等前端或全栈前端项目。

```text
src/
├─ app/
│  ├─ core/
│  │  ├─ config/
│  │  ├─ auth/
│  │  ├─ api/
│  │  └─ guards/
│  ├─ shared/
│  │  ├─ components/
│  │  ├─ pipes/
│  │  ├─ utils/
│  │  └─ types/
│  ├─ features/
│  │  ├─ orders/
│  │  ├─ products/
│  │  ├─ reports/
│  │  ├─ training/
│  │  └─ knowledge-base/
│  └─ layouts/
├─ assets/
└─ environments/
```

## 单个业务模块结构

每个业务模块建议保持同一结构。

```text
features/某业务模块/
├─ README.md
├─ pages/
├─ components/
├─ services/
├─ rules/
├─ types/
├─ utils/
└─ tests/
```

各目录含义：

| 目录 | 放什么 | AI 修改原则 |
|---|---|---|
| `README.md` | 模块目标、边界、主要文件、注意事项 | 修改前先读 |
| `pages/` | 页面级组件 | 只处理页面展示和交互编排 |
| `components/` | 可复用 UI 组件 | 不写复杂业务规则 |
| `services/` | 数据访问、接口调用、业务服务 | 不直接写 UI |
| `rules/` | 业务判断规则 | 优先独立测试 |
| `types/` | 类型、字段、数据结构 | 新字段先改这里 |
| `utils/` | 无状态工具函数 | 不依赖页面状态 |
| `tests/` | 测试或验证样例 | 关键规则必须覆盖 |

## 推荐非应用工作流结构

如果还没到搭应用阶段，只是脚本、报告或自动化工作流，使用轻量结构：

```text
workflow-name/
├─ docs/
│  ├─ 规则说明.md
│  ├─ 输入材料说明.md
│  └─ 输出报告模板.md
├─ scripts/
├─ samples/
├─ outputs/
└─ README.md
```

## 代码分层规则

### 页面层

页面层只负责：

- 展示数据。
- 接收用户操作。
- 调用服务。
- 显示加载、错误、空状态。

页面层不应该：

- 直接写复杂业务判断。
- 直接拼数据库查询。
- 直接处理大量数据清洗。
- 直接调用多个底层 API。

### 服务层

服务层负责：

- 调用 API / SDK / 数据库。
- 组合业务操作。
- 处理数据保存、查询、更新、删除。
- 隔离第三方服务变化。

### 规则层

规则层负责：

- 审核规则。
- 评分规则。
- 分类规则。
- 字段归并规则。
- 风险判断规则。

规则层应尽量是纯函数，方便 AI 单独修改和测试。

### 类型层

类型层负责：

- 定义实体字段。
- 定义输入输出结构。
- 定义状态枚举。
- 定义报告结果结构。

AI 不得在页面中随意新增字段。新增字段必须先更新类型。

## 命名规范

推荐按业务语义命名。

示例：

```text
OrderReviewResult
OrderAttachmentCheck
ProductSkuGroup
TrainingScore
KnowledgeItem
ReportGenerationTask
```

避免：

```text
DataItem
InfoObj
TempService
CommonHelper
NewPage
```

## AI 修改代码前检查

AI 写代码前必须先确认：

1. 当前任务属于哪个业务模块。
2. 是否已有模块 README。
3. 是否已有类型定义。
4. 是否已有业务规则。
5. 是否已有服务封装。
6. 是否需要新增测试或验证样例。
7. 是否会影响其他模块。

如果以上信息缺失，先补文档或任务卡，不直接写代码。

## 禁止事项

- 不把业务规则写死在按钮点击事件里。
- 不让页面直接访问多个底层接口。
- 不在多个模块复制同一套规则。
- 不新增含义模糊的 `utils/common/helper`。
- 不在没有类型定义时随意造字段。
- 不在一个提交或一次任务中跨多个业务模块大范围修改。
- 不为了“看起来高级”引入过度抽象。

## 应用化前置条件

从 skill 进入应用化前，应满足：

1. 已有明确业务流程。
2. 已有输入材料和输出结果。
3. 已有基础规则或判断标准。
4. 已经通过至少一次真实任务验证。
5. 已登记到 `12_app_candidates/应用化候选清单.md`。
6. 已确认代码架构类型：轻量工作流 / 前端应用 / 全栈应用。

## 验收标准

一个 AI 友好代码项目应满足：

1. 目录按业务模块组织。
2. 每个核心模块有 README。
3. 每个核心模块有 types、services、rules 的明确位置。
4. 页面层不直接写复杂业务规则。
5. 数据访问统一封装。
6. 关键业务规则可单独测试或验证。
7. AI 可以根据模块路径定位修改范围。
8. 新增功能有任务卡、规则或需求说明支撑。
