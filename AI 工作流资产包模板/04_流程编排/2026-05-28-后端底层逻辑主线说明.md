# 后端底层逻辑主线说明

## 固定主线

当前后端底层逻辑按以下顺序维护：

```text
组织 -> 员工 -> 考勤 -> 薪酬
```

这个顺序不是 UI 菜单顺序，而是数据依赖顺序。后续改代码时，必须先确认上游域的输出契约，再改下游域。

## 基础层

| 内容 | 位置 | 说明 |
|---|---|---|
| 后端基础层 | `server/core/foundation.mjs` | 运行配置、数据目录、JSON Store、业务主线 |
| 主服务 | `server/data-server.mjs` | 现有业务函数和 API 路由 |
| 健康检查 | `/api/health` | 返回运行目录和 `domainFlow` |
| 迁移包 | `/api/hr-core/migration-bundle` | 按组织、员工、考勤、薪酬顺序导出 |

## 四个 Foundation 接口

| 顺序 | 域 | 接口 | 作用 |
|---|---|---|---|
| 1 | 组织 | `/api/organization-management/foundation` | 组织结构、岗位、职级、组织设置的来源、写入规则、员工输出契约 |
| 2 | 员工 | `/api/employee-management/foundation` | 员工主数据、组织/岗位/职级关联、考勤和薪酬输出契约 |
| 3 | 考勤 | `/api/attendance-management/foundation` | 考勤人员、设置、排班、打卡、日/月统计、假勤外勤和薪酬输出契约 |
| 4 | 薪酬 | `/api/payroll-management/foundation` | 薪酬可消费数据边界、统计项、外部数据、公式追踪契约 |

## 数据边界

### 组织

- 组织结构按 `code` 写入和覆盖。
- 岗位按 `code || name` 写入和覆盖。
- 职级按 `code || name` 写入和覆盖。
- 输出给员工的匹配口径是：
  - 组织：`fullPath`、`name`
  - 岗位：`name`
  - 职级：`code`、`name`

### 员工

- 员工号是员工域主键。
- 员工主数据先读员工花名册，再用本地已入职员工按员工号覆盖。
- 员工输出给考勤的必要字段：
  - `employeeNo`
  - `name`
  - `department`
  - `deptFullPath`
  - `attendanceGroupName`
  - `shiftName`
  - `employeeStatus`
- 员工输出给薪酬的必要字段：
  - `employeeNo`
  - `name`
  - `department`
  - `deptFullPath`
  - `position`
  - `rankCode`
  - `rankName`

### 考勤

- 考勤域只能用员工号识别人员。
- 姓名、部门、岗位、考勤组、班次只作为计算和展示口径。
- 考勤月汇总输出给薪酬的字段包括：
  - `empId`
  - `name`
  - `dept`
  - `position`
  - `deptFullPath`
  - `shouldWorkDays`
  - `actualWorkDays`
  - `absentDays`
  - `lateMinutes`
  - `normalHours`
  - `customMetrics`

### 薪酬

- 薪酬只能消费员工主数据、考勤月汇总、统计项和外部数据。
- 薪酬不能直接读取原始打卡明细作为正式输入。
- 薪酬不能用姓名匹配人员。
- 当前 `src/app/pages/SalaryCalculation.tsx` 仍有本地样例数据和本地公式，后续迁移必须接入后端薪酬输入边界。
- 后续公式追踪结构固定为：
  - `expression`
  - `inputs`
  - `blockers`
  - `notes`

## 后续维护规则

1. 改组织前，先看 `/api/organization-management/foundation`。
2. 改员工前，先看组织 foundation 和 `/api/employee-management/foundation`。
3. 改考勤前，先看员工 foundation 和 `/api/attendance-management/foundation`。
4. 改薪酬前，先看员工、考勤 foundation 和 `/api/payroll-management/foundation`。
5. 不要绕过上游域直接读散落字段。
6. 不要为了整理而拆散文件；只有当一块逻辑已经稳定、边界清楚时再考虑抽出模块。
