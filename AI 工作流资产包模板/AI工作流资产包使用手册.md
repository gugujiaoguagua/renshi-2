# AI 工作流资产包使用手册

## 1. 这是什么

AI 工作流资产包是一套让项目经验持续沉淀的文件夹模板。

它的目的不是保存一堆提示词，而是让一个新项目从开始到结束都能持续积累：

- 项目规则
- 任务卡
- 业务流程
- AI 执行边界
- 可复用模板
- skills 技能包
- 每次对话收集
- 每日总结
- 项目完成收口
- 应用化候选
- AI 友好代码架构

最终目标是：项目完成时，不只是交付了结果，还留下可复用的 skill、规则、案例和流程资产。

## 2. 新项目怎么开始

复制整个 `AI 工作流资产包模板` 文件夹到新项目目录，并重命名为：

```text
AI工作流资产包
```

然后先填写 3 个文件：

1. `01_项目说明.md`
2. `02_角色与目标.md`
3. `03_规则与约束/01_项目硬规则.md`

第一次不要填满所有目录。先跑起来。

## 3. 最小使用方式

新项目初期只需要用 6 个核心入口：

| 文件或目录 | 作用 |
|---|---|
| `00_使用说明.md` | AI 启动入口 |
| `01_项目说明.md` | 项目背景、资料位置、成功标准 |
| `02_角色与目标.md` | AI 角色、边界、目标 |
| `03_规则与约束/` | 硬规则和规则升级 |
| `10_tasks/` | 每个任务一张任务卡 |
| `09_memory/` | 沉淀规则、偏好、案例、每日总结 |

其他目录不用急着填：

- 有重复流程时，再用 `04_流程编排/`
- 有可复用能力时，再用 `05_skills/`
- 输出格式稳定后，再用 `06_templates/`
- 有脚本工具时，再用 `07_tools/`
- 有典型样例时，再用 `08_examples/`
- 需要验收时，再用 `11_eval/`
- 高频多人使用时，再用 `12_app_candidates/`

## 4. 每次开始任务时怎么说

最简单的启动语：

```text
请读取本项目的 AI工作流资产包，先按最小使用方式执行。请为本次任务创建任务卡，完成任务后收集有用信息并更新 memory。
```

如果希望完整走流程：

```text
请读取本项目的 AI工作流资产包，按完整流程执行：创建任务卡、判断任务类型、调用或创建 skill、完成交付、质量检查、复盘、更新 memory/rules/skills/app candidates。
```

如果只想讨论，不希望 AI 写文件：

```text
先只讨论，不写文件。请基于 AI工作流资产包的规则，告诉我这个任务应该怎么拆、适合做 skill 还是应用。
```

如果只想执行任务，不更新资产包：

```text
请只按 AI工作流资产包的规则执行本次任务，不要更新资产包文件。
```

## 5. 每个任务怎么流转

标准流程：

```text
用户提出任务
→ AI 读取资产包
→ 创建任务卡
→ 判断任务类型
→ 执行任务
→ 输出交付物
→ 质量检查
→ 对话结束收集
→ 任务复盘
→ 更新 memory / rules / skills / app candidates
```

任务类型分三类：

| 类型 | 处理方式 |
|---|---|
| 临时任务 | 直接完成，结束后记录经验 |
| 可复用任务 | 创建或升级 skill |
| 高频复杂任务 | 先用 skill 跑通，再进入应用化候选 |

## 6. 每次对话结束要收集什么

每次对话结束前，AI 应检查是否有值得沉淀的信息。

需要收集：

- 新业务规则
- 用户偏好
- 可复用输出格式
- AI 误判或风险点
- 可沉淀为 skill 的流程
- 可加入 examples 的输入输出样例
- 可能应用化的高频复杂需求

使用模板：

```text
06_templates/对话结束收集模板.md
```

默认写入：

```text
09_memory/
```

重要原则：第一次出现的信息只进 memory，不直接升级为正式规则。

## 7. 每天结束怎么总结

当一天工作结束时，对 AI 说：

```text
请根据今天的任务和对话，使用每日总结模板，更新本项目的每日总结、规则沉淀、失败案例、成功案例和 skill 成熟度跟踪。
```

使用模板：

```text
06_templates/每日总结模板.md
```

主要更新：

```text
09_memory/05_每日总结.md
09_memory/01_业务规则沉淀.md
09_memory/03_失败案例.md
09_memory/04_成功案例.md
09_memory/06_skill成熟度跟踪.md
```

每日总结不是普通日报，它要回答：

- 今天推进了什么？
- 今天沉淀了什么？
- 哪些只是观察，不能升级？
- 哪些 skill 变成熟？
- 明天优先做什么？

## 8. 什么时候升级规则

规则升级遵守三层沉淀法：

| 阶段 | 动作 |
|---|---|
| 第一次出现 | 记录到 memory |
| 第二次验证 | 进入规则候选或 skill 候选 |
| 第三次稳定 | 升级到规则总表、流程文件或 `SKILL.md` |

不要因为一次任务就修改正式规则。这样可以避免把偶然经验固化成错误标准。

## 9. 什么时候创建 skill

适合创建 skill 的情况：

- 同类任务已经出现 2 次以上。
- 输入材料和输出结果比较稳定。
- 任务主要靠 AI 判断、整理、生成、分析。
- 不需要登录、权限、复杂界面或长期状态流转。

skill 标准目录：

```text
05_skills/某个技能/
├─ SKILL.md
├─ references/
├─ examples/
└─ scripts/
```

最重要的是 `SKILL.md`。它必须包含：

- `name`
- `description`
- 适用场景
- 不适用场景
- 输入材料
- 执行流程
- 输出格式
- 质量检查
- 升级规则

## 10. skill 成熟度怎么判断

成熟度记录在：

```text
09_memory/06_skill成熟度跟踪.md
```

等级如下：

| 等级 | 含义 |
|---|---|
| L0 | 临时提示词，只解决一次任务 |
| L1 | skill 草稿，有初步流程 |
| L2 | 可复用 skill，被 2 次以上任务验证 |
| L3 | 可导入 skill，有标准 `SKILL.md` |
| L4 | 带样例 skill，有 examples 和质量检查 |
| L5 | 可应用化，高频多人使用，值得做 demo |

项目结束时，至少要判断哪些 skill 达到 L3。达到 L3 的 skill 才算真正可以给 Claude / Codex 使用。

## 11. 什么时候考虑做应用

不要一开始就做应用。先用 skill 跑通。

满足以下情况，再进入应用化候选：

- 多人使用
- 高频重复
- 需要登录权限
- 需要长期保存数据
- 需要审批或状态流转
- 需要手机端或网页端
- 人工启动 skill 成本太高

登记位置：

```text
12_app_candidates/应用化候选清单.md
```

进入 demo 或应用开发前，必须先读取：

```text
03_规则与约束/05_AI友好代码架构规范.md
```

并使用：

```text
06_templates/代码项目架构模板.md
```

先把代码结构定清楚，再让 AI 写代码。

## 12. 代码项目怎么启动

当一个流程已经决定进入 demo、小工具或应用开发时，不要直接让 AI 写页面。先让 AI 定架构。

启动语：

```text
这个流程准备进入 demo 开发。请读取 AI友好代码架构规范，并使用代码项目架构模板，先帮我确定项目类型、目录结构、业务模块、types、services、rules 和验收方式，暂时不要写代码。
```

如果已经可以写代码：

```text
请按 AI友好代码架构规范搭建最小 demo。每次只修改一个业务模块，新增字段先更新 types，业务规则放入 rules，数据访问放入 services，完成后按代码架构验收清单自检。
```

推荐判断：

| 情况 | 做法 |
|---|---|
| 只是一次性处理文件 | 不做应用，使用 skill 或脚本 |
| 流程可复用但少数人用 | 做 skill |
| 流程稳定且需要自动执行 | 做轻量工作流 |
| 多人高频且需要界面 | 做 demo 应用 |
| 需要权限、数据、状态流转 | 做正式应用候选 |

代码开发必须遵守：

- 按业务模块组织目录。
- 页面不写复杂业务规则。
- 业务规则放入 `rules/`。
- 数据访问放入 `services/` 或 `api/`。
- 新字段先更新 `types/`。
- 核心模块必须有 README。
- AI 修改代码前先确认修改范围。

## 13. 项目完成时怎么收口

项目结束时，对 AI 说：

```text
请使用项目完成收口模板，对本项目的任务、规则、memory、skills、examples 和应用化候选做最终收口，输出可使用的 skill 包清单。
```

使用模板：

```text
06_templates/项目完成收口模板.md
```

项目完成时必须确认：

- 所有任务卡已完成、暂停或归档。
- 有最终规则清单。
- 有最终 skill 清单。
- 成熟 skill 是否达到 L3。
- 成功案例是否进入 examples 或 memory。
- 高频流程是否进入应用化候选。

## 14. 常用指令

### 初始化项目

```text
请读取 AI工作流资产包/00_使用说明.md，帮我初始化这个项目，只填写必要文件，不要展开所有目录。
```

### 创建任务

```text
请根据任务卡模板，为这个任务创建任务卡，并判断它是临时任务、skill 任务还是应用化候选任务。
```

### 对话结束收集

```text
请对本轮对话做轻量收集，只记录确定有价值的信息，更新 memory，不要随便升级正式规则。
```

### 每日总结

```text
请做今天的每日总结，更新每日总结、业务规则沉淀、失败案例、成功案例和 skill 成熟度跟踪。
```

### 升级 skill

```text
请检查这个任务是否已经适合升级为 skill。如果适合，先创建 skill 草稿，不要直接定为正式版本。
```

### 项目完成收口

```text
请对项目做完成收口，输出最终规则清单、成熟 skill 清单、可复用案例和应用化候选。
```

### 代码架构规划

```text
请读取 AI友好代码架构规范，使用代码项目架构模板，先规划这个 demo 的代码结构，不要直接写代码。
```

### 代码开发验收

```text
请按代码架构验收标准检查当前项目：目录是否按业务模块组织，规则是否脱离 UI，数据访问是否封装，types 是否完整，AI 后续是否能局部修改。
```

## 15. 验收这个资产包是否用起来了

一个项目真正用起来，应满足：

1. 有项目说明。
2. 有角色与目标。
3. 有任务卡。
4. 每次对话有信息收集。
5. 每天结束有总结。
6. 有业务规则沉淀。
7. 有失败案例和成功案例。
8. 有 skill 成熟度跟踪。
9. 项目结束有收口报告。
10. 至少一个 skill 达到 L3，或说明为什么还没有成熟。

如果项目进入代码开发，还应满足：

1. 已有代码项目架构模板。
2. 已按业务模块规划目录。
3. 业务规则、类型、服务、页面分层清楚。
4. 有代码架构验收记录。

## 16. 使用原则

记住三句话：

```text
先跑通，不填满。
先进 memory，不急着改规则。
项目结束时，skill 也要能用。
```
